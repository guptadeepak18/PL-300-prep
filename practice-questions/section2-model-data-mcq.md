## Question 1

**A BI developer is designing a Power BI data model for a retail company. The model will contain a fact table "Sales" and dimension tables for Date, Product, Customer, and Store. Which schema design best supports this model for optimal query performance in Power BI?**

A) Snowflake schema — normalise the dimension tables further into sub-dimensions  
B) Star schema — keep dimension tables flat and denormalised around the central fact table  
C) Third Normal Form (3NF) — preserve all referential relationships from the source OLTP system  
D) Galaxy schema — use multiple fact tables each with their own dimension tables  

**✅ Correct Answer: B**

**Explanation:** A **star schema** is the recommended design for Power BI data models. Dimension tables are flat and denormalised (all attributes in one table), joined directly to the central fact table. The VertiPaq engine is highly optimised for this pattern, and DAX filter propagation along single relationships works cleanly. A **snowflake schema** (A) introduces additional relationships and join hops, complicating DAX and reducing performance. **3NF** (C) comes from OLTP design and has many small tables with complex joins — not suitable for analytical queries. A **galaxy schema** (D) applies when there are multiple fact tables, which is a valid extension of the star schema but not the primary consideration here.

## Question 2

**A "Sales" fact table contains 50 million rows. A developer notices that it includes columns "ProductName", "ProductCategory", and "ProductSubcategory" that are repeated for every row. What is the recommended action to optimise the data model?**

A) Remove those columns from the model entirely  
B) Move those columns to a separate "Product" dimension table and create a relationship on "ProductID"  
C) Create calculated columns in DAX to derive the product attributes  
D) Convert the fact table to DirectQuery mode  

**✅ Correct Answer: B**

**Explanation:** Storing repeated text attributes (ProductName, ProductCategory, ProductSubcategory) in a 50-million-row fact table severely increases the model's memory footprint. Moving these columns to a **separate dimension table** reduces cardinality in the fact table to just a numeric `ProductID` key, which compresses extremely well in VertiPaq. The dimension table is tiny by comparison. Removing them (A) loses useful report attributes. Creating DAX calculated columns (C) materialises the data back into the fact table — same storage problem. DirectQuery (D) avoids storage but doesn't resolve the poor model design.

## Question 3

**In a star schema, what is the primary purpose of a surrogate key in a dimension table?**

A) To store the business key from the source system for user display  
B) To serve as a unique integer identifier generated by the warehouse, independent of the business key  
C) To create a composite primary key by combining multiple columns  
D) To enable DirectQuery mode on the dimension table  

**✅ Correct Answer: B**

**Explanation:** A **surrogate key** is a system-generated integer (typically auto-incremented) used as the primary key of a dimension table, independent of any business meaning. It allows the dimension table to handle slowly changing dimensions (SCDs), late-arriving data, and source system changes without breaking relationships. The **business key** (A) (e.g., CustomerCode from the source system) is stored separately for matching/display purposes. A surrogate key is a single column, not a composite key (C). Storage mode (D) is unrelated to keys.

## Question 4

**A data model has a "Sales" fact table related to a "Date" dimension and a "Product" dimension. There is also a "ProductCategory" table related to "Product". This forms a chain: Sales → Product → ProductCategory. What schema pattern does this represent, and what is the recommended Power BI approach?**

A) Star schema — no change needed  
B) Snowflake schema — recommended to flatten ProductCategory into the Product dimension table  
C) Galaxy schema — add a separate fact table for categories  
D) Hybrid schema — acceptable as long as bidirectional filtering is enabled  

**✅ Correct Answer: B**

**Explanation:** A chain where Sales → Product → ProductCategory is a **snowflake schema** — the Product dimension is further normalised into a sub-dimension. While Power BI can handle this, the recommended best practice is to **flatten** the ProductCategory attributes (CategoryName, etc.) directly into the Product dimension table. This simplifies the model, eliminates an extra relationship hop, and improves DAX performance and usability. Bidirectional filtering (D) does not compensate for the performance and complexity costs of snowflaking.

## Question 5

**Which of the following best describes a "fact table" in a star schema data model?**

A) A table containing descriptive attributes used to filter and group data  
B) A table containing quantitative, measurable events or transactions, typically with foreign keys to dimension tables  
C) A table that stores only calculated DAX measures  
D) A table used exclusively for date and time dimensions  

**✅ Correct Answer: B**

**Explanation:** A **fact table** stores measurable, quantitative data about business events (sales transactions, inventory movements, web page views) and includes **foreign keys** that reference dimension tables. Metrics like Revenue, Quantity, and Cost live in fact tables. **Dimension tables** (A) hold descriptive attributes (customer name, product category, date labels). Measures are not stored in tables — they are DAX expressions (C). Date/time tables are a specific type of dimension table (D).

## Question 6

**A developer needs to model a scenario where a "Sales" fact table needs to link to the "Date" dimension in three different ways: OrderDate, ShipDate, and DueDate. What is the recommended approach in Power BI?**

A) Create three separate Date dimension tables (one per date role) and connect each to the Sales table  
B) Use a single Date table with active relationship on OrderDate; create inactive relationships for ShipDate and DueDate, using USERELATIONSHIP in measures  
C) Add OrderDate, ShipDate, and DueDate columns directly to the Sales fact table without any relationship  
D) Create a junction table linking Sales to Date  

**✅ Correct Answer: B**

**Explanation:** The recommended approach uses a single Date table with one **active relationship** (OrderDate) and additional **inactive relationships** (ShipDate, DueDate). When writing DAX measures that need to filter by ShipDate or DueDate, you use the `USERELATIONSHIP` function inside `CALCULATE` to activate the appropriate relationship. Option A (multiple Date tables) works but violates the DRY principle and is harder to maintain. Option C loses all time intelligence functionality. Option D adds unnecessary complexity.

## Question 7

**In Power BI's data model view, a developer sees a relationship line with a "1" on one side and a "*" on the other. What does this represent?**

A) A many-to-many relationship  
B) A one-to-one relationship  
C) A one-to-many relationship, where the "1" side is the dimension and the "*" side is the fact table  
D) An inactive relationship  

**✅ Correct Answer: C**

**Explanation:** In Power BI's model view, a relationship line shows **"1"** on the unique-key side (typically the dimension table, where each value appears exactly once) and **"*"** on the foreign-key side (typically the fact table, where that key value can appear many times). This is a **one-to-many (1:M)** relationship. Many-to-many shows "*" on both sides (A). One-to-one shows "1" on both sides (B). Inactive relationships are shown as a dashed line, not indicated by cardinality symbols alone (D).

## Question 8

**A Power BI model has a "Sales" table and a "Returns" table. Both tables contain a "ProductID" column that relates to a "Products" dimension. The developer creates relationships: Sales → Products (active) and Returns → Products (active). Filters on Products flow into Sales and Returns. What cross-filter direction should be set on both relationships for standard star schema filtering?**

A) Bidirectional on both relationships  
B) Single direction (from Products to Sales, and from Products to Returns)  
C) Single direction (from Sales to Products, and from Returns to Products)  
D) Bidirectional on Sales → Products; Single on Returns → Products  

**✅ Correct Answer: B**

**Explanation:** In a star schema, filter direction flows **from the "one" side (dimension) to the "many" side (fact)**. Slicers and report filters applied to Products should flow down into both Sales and Returns. Setting cross-filter direction as **Single: from Products (1) to Sales/Returns (*)** achieves this. Bidirectional (A) can cause ambiguity, circular filter paths, and performance issues and should be used only when needed. Options C and D have the direction reversed or inconsistent.

## Question 9

**When should a developer use bidirectional cross-filter direction on a relationship in Power BI?**

A) Always — bidirectional is the most powerful and should be the default  
B) Never — bidirectional causes performance problems in all cases  
C) When a filter must propagate from the "many" side to the "one" side (e.g., to filter a dimension by activity in the fact table), such as in many-to-many bridge table patterns  
D) Only in DirectQuery mode  

**✅ Correct Answer: C**

**Explanation:** **Bidirectional cross-filtering** allows filters to flow in both directions across a relationship. It is appropriate in specific scenarios such as many-to-many relationships via bridge tables, or when you need a dimension to be filtered based on values in the fact table. However, it should not be the default (A) as it can introduce ambiguity, unexpected filter propagation, and performance degradation. It is not entirely forbidden (B), and it works in both Import and DirectQuery modes (D is wrong).

## Question 10

**A model contains a "Products" table and a "Sales" table with a one-to-many relationship on ProductID. A developer creates a measure `[SalesCount] = COUNTROWS(Sales)`. When a product slicer is used, how does filter context propagate?**

A) The slicer filters the Sales table directly by joining on ProductID in DAX  
B) The slicer applies a filter on the Products table, which propagates via the relationship to the Sales table, filtering rows in Sales to those matching the selected ProductID(s)  
C) The slicer has no effect on the measure unless CALCULATE is used  
D) The measure must explicitly reference the Products table to respond to the slicer  

**✅ Correct Answer: B**

**Explanation:** When a user selects a product in a slicer, a **filter context** is applied to the Products table. This filter propagates along the active relationship (from Products "1" side to Sales "*" side), restricting the Sales rows visible to those with matching ProductIDs. `COUNTROWS(Sales)` then counts only those filtered rows. This propagation is automatic in Power BI — no explicit CALCULATE or Products reference is needed in the measure (A, C, D are incorrect).

## Question 11

**A developer has a "Employees" table and a "Manager" column in the same table (self-referencing hierarchy: each employee has a ManagerID that points to another EmployeeID). Power BI does not support self-referencing (recursive) relationships. What is the recommended workaround?**

A) Use DAX PATH() and PATHITEM() functions to handle the hierarchy without a relationship  
B) Create a separate "Managers" table as a duplicate of Employees and create a relationship between Employees and Managers  
C) Use bidirectional filtering on the self-referencing relationship  
D) Flatten the hierarchy in Power Query to a maximum of 5 levels  

**✅ Correct Answer: A**

**Explanation:** Power BI does not support self-referencing relationships. The recommended solution is to use DAX **`PATH()`** and **`PATHITEM()`** functions (along with `PATHLENGTH()`, `PATHCONTAINS()`) to handle parent-child hierarchies within a single table. `PATH()` generates a delimited string of all ancestor IDs for each employee, which `PATHITEM()` can then parse to flatten hierarchy levels. Creating a duplicate Managers table (B) works as a modelling workaround but is complex. Bidirectional filtering (C) does not resolve the self-reference limitation. Flattening in Power Query (D) is another approach but hardcodes the level count and loses flexibility.

## Question 12

**A "Sales" fact table has a many-to-many relationship with a "Promotions" table (one sale can have multiple promotions; one promotion applies to multiple sales). How should this be modelled in Power BI?**

A) Create a direct many-to-many relationship between Sales and Promotions using Power BI's built-in M:M cardinality  
B) Create a bridge (junction) table "SalesPromotions" with foreign keys to both Sales and Promotions, and create one-to-many relationships from each dimension to the bridge table  
C) Store the promotion data as a JSON column within the Sales fact table  
D) Use bidirectional relationships between Sales and Promotions  

**✅ Correct Answer: B**

**Explanation:** While Power BI does support **direct many-to-many relationships** (A), the preferred approach for a true M:M scenario between a fact and another entity is to use a **bridge (junction) table** — "SalesPromotions" containing SaleID and PromotionID — with standard 1:M relationships from Sales and Promotions to the bridge. This is more transparent, maintainable, and avoids the ambiguity of direct M:M with bidirectional filtering. Option A can work but requires careful filter direction management. Option C is an anti-pattern. Option D alone does not resolve M:M cardinality.

## Question 13

**What is the risk of using many-to-many (M:M) relationships with bidirectional cross-filtering in Power BI?**

A) M:M relationships are not supported at all in Power BI  
B) Bidirectional M:M relationships can cause ambiguous filter propagation, unexpected double-counting in measures, and performance degradation  
C) M:M relationships only work in DirectQuery mode  
D) Bidirectional filtering on M:M relationships requires DAX CROSSJOIN  

**✅ Correct Answer: B**

**Explanation:** Many-to-many relationships with bidirectional cross-filtering are supported in Power BI but carry significant risks: filters can propagate in unexpected directions, causing **double-counting** in aggregated measures, unintended cross-table filtering, and query performance issues. These relationships require careful testing and often explicit DAX to control filter context. Power BI does support M:M (A is wrong). They work in both Import and DirectQuery (C is wrong). CROSSJOIN is a DAX function for Cartesian products, unrelated to relationship filters (D is wrong).

## Question 14

**A developer activates an inactive relationship using USERELATIONSHIP inside CALCULATE. What is important to understand about USERELATIONSHIP?**

A) It permanently makes the relationship active in the model  
B) It activates the specified relationship only within the scope of the CALCULATE expression in which it is used, deactivating the currently active relationship for that same column pair during that evaluation  
C) It can only be used with date/time columns  
D) It requires bidirectional cross-filter to be enabled  

**✅ Correct Answer: B**

**Explanation:** `USERELATIONSHIP(columnA, columnB)` is a modifier function used inside `CALCULATE`. It **temporarily activates** the specified inactive relationship only during the evaluation of that `CALCULATE` expression. The currently active relationship between those two tables is temporarily deactivated for that evaluation. After the `CALCULATE` expression completes, filter context reverts to normal. It does not change the permanent model configuration (A). It is not limited to date columns (C), though that is its most common use case. It does not require bidirectional filtering (D).

## Question 15

**A Power BI model has a "Customers" dimension and a "Sales" fact table. The relationship cardinality is set as 1:M (Customers to Sales). A developer accidentally loads duplicate CustomerIDs into the Customers table. What will happen to the relationship?**

A) The relationship will automatically convert to M:M  
B) The relationship will show a warning/error indicator in the model view because the "1" side must have unique values  
C) The duplicate values will be silently ignored  
D) The fact table will also be deduplicated automatically  

**✅ Correct Answer: B**

**Explanation:** A **one-to-many relationship requires the "one" side to have unique (non-duplicate) values** in the join column. If duplicates exist in Customers.CustomerID, Power BI will display a **warning** on the relationship (yellow triangle indicator) and the relationship may behave as many-to-many, returning incorrect or unexpected results. The model will not automatically convert the cardinality (A) or silently ignore duplicates (C). The fact table is unaffected (D).

## Question 16

**What does a "role-playing dimension" mean in the context of a Power BI data model?**

A) A dimension table that has row-level security roles assigned  
B) A single dimension table (typically Date) that is related to a fact table via multiple relationships, each serving a different analytical role  
C) A dimension used only in DirectQuery mode  
D) A dimension shared across multiple datasets  

**✅ Correct Answer: B**

The classic example is a Date table that relates to Sales on OrderDate (active), ShipDate (inactive), and DueDate (inactive). The same Date dimension "plays" different roles depending on which relationship is being used. It has nothing to do with RLS roles (A), DirectQuery (C), or cross-dataset sharing (D).

## Question 17

**In Power BI, can two active relationships exist between the same pair of tables simultaneously?**

A) Yes, if both use different join columns  
B) Yes, if one is set to bidirectional  
C) No — only one active relationship can exist between any two tables; additional relationships must be inactive  
D) Yes, in DirectQuery mode only

If you need to relate two tables via multiple columns (e.g., Sales to Date on both OrderDate and ShipDate), only one can be active; the others must be **inactive**. Inactive relationships are activated on demand using `USERELATIONSHIP`. This rule applies regardless of join columns (A), filter direction (B), or storage mode (D).

## Question 19

**Which of the following is a requirement for a table to be marked as a "Date table" in Power BI?**

A) The table must be named "Date" or "Calendar"  
B) The table must contain a date column with no gaps and no duplicate dates, covering the full range of dates present in the fact tables  
C) The table must be created using the CALENDAR() DAX function  
D) The table must include at least Year, Quarter, Month, and Day columns  

**✅ Correct Answer: B**

**Explanation:** To mark a table as a Date table in Power BI (via the Model view or right-click menu), the designated date column must: (1) contain **unique values** (no duplicates), (2) have **no gaps** — every single date in the range must be present, and (3) be of **Date data type**. The table does not need to be named anything specific (A). It can be created with CALENDAR(), CALENDARAUTO(), Power Query, or imported from a source (C is not required). Additional columns like Year, Quarter, Month are helpful but not required to mark the table as a Date table (D).

## Question 20

**What is the difference between CALENDAR() and CALENDARAUTO() in DAX?**

A) CALENDAR() generates a date range based on dates already in the model; CALENDARAUTO() requires explicit start and end date arguments  
B) CALENDAR() requires explicit start and end date arguments; CALENDARAUTO() scans the model for the minimum and maximum dates across all date/datetime columns and generates the range automatically  
C) CALENDAR() only generates monthly intervals; CALENDARAUTO() generates daily intervals  
D) CALENDARAUTO() is only available in Power BI Premium  

**✅ Correct Answer: B**

**Explanation:** `CALENDAR(start_date, end_date)` generates a single-column table of dates between the two **explicitly specified** dates. `CALENDARAUTO([fiscal_year_end_month])` automatically scans **all date and datetime columns** in the model (excluding calculated tables) to find the minimum and maximum values and generates a contiguous date table covering that range. A has the definitions backwards. Both generate daily (not monthly) intervals (C). CALENDARAUTO() is available in all Power BI tiers (D is wrong).

## Question 21

**A developer writes the DAX measure `[YTD Sales] = TOTALYTD([Total Sales], 'Date'[Date])`. What does TOTALYTD do?**

A) Calculates Total Sales for the previous year  
B) Calculates the cumulative sum of Total Sales from the start of the current year to the last date in the current filter context  
C) Calculates the year-over-year percentage growth of Total Sales  
D) Calculates Total Sales for the current quarter  

**✅ Correct Answer: B**

**Explanation:** `TOTALYTD(expression, dates)` is a time intelligence function that evaluates the expression (here, `[Total Sales]`) over a date range from **January 1 of the current year** (or fiscal year start if specified) through to the **last date in the current filter context**. It produces a running year-to-date total. Previous year (A) would use `SAMEPERIODLASTYEAR`. Year-over-year growth (C) requires dividing current by prior year. Quarter-to-date (D) uses `TOTALQTD`.

## Question 22

**What does `SAMEPERIODLASTYEAR('Date'[Date])` return when the current filter context is March 2024?**

A) The dates of March 2023  
B) The dates of Q1 2023  
C) The dates of 2023 year-to-date  
D) The date March 2025  

**✅ Correct Answer: A**

**Explanation:** `SAMEPERIODLASTYEAR` returns a table of dates corresponding to the **same period in the previous year**. If the current filter context covers March 2024, it returns the dates in March 2023. It is a date-shifting function commonly used in `CALCULATE` to compare the same calendar period one year back. It does not expand to an entire quarter (B), year-to-date (C), or shift forward (D).

## Question 23

**A developer wants to calculate sales for the previous month, regardless of what the current month's filter context is. Which DAX function achieves this?**

A) `DATEADD('Date'[Date], -1, MONTH)`  
B) `PREVIOUSMONTH('Date'[Date])`  
C) `PARALLELPERIOD('Date'[Date], -1, MONTH)`  
D) Both A and B — they are equivalent  

**✅ Correct Answer: B**

**Explanation:** `PREVIOUSMONTH('Date'[Date])` returns the complete set of dates for the **entire month prior** to the first month in the current filter context — ideal for "previous month total" measures. `DATEADD('Date'[Date], -1, MONTH)` (A) shifts each date in the current filter back by one month — if the filter covers partial months, DATEADD returns partial months too. `PARALLELPERIOD('Date'[Date], -1, MONTH)` (C) returns all dates for the full prior month starting from the first date in context, similar to PREVIOUSMONTH. B is specifically designed for "previous full month" semantics.

## Question 24

**Which DAX time intelligence function would you use to compare a measure against the same quarter one year ago?**

A) `TOTALQTD`  
B) `PREVIOUSQUARTER`  
C) `DATEADD('Date'[Date], -1, YEAR)` combined with `TOTALQTD`  
D) `PARALLELPERIOD('Date'[Date], -4, QUARTER)`  

**✅ Correct Answer: D**

**Explanation:** `PARALLELPERIOD('Date'[Date], -4, QUARTER)` shifts the current quarter's date range back by 4 quarters (equivalent to 1 year ago, same quarter). This returns the same full quarter from the prior year. `TOTALQTD` (A) calculates quarter-to-date for the current quarter, not a comparison period. `PREVIOUSQUARTER` (B) returns the immediately preceding quarter, not the same quarter last year. Option C combines two functions incorrectly for this purpose.

## Question 25

**A developer uses `DATEADD('Date'[Date], -1, YEAR)` inside a CALCULATE measure. If the current filter selects April and May 2024, what dates does DATEADD return?**

A) April and May 2023  
B) March and April 2024  
C) The full year 2023

If the current context includes April and May 2024, `DATEADD('Date'[Date], -1, YEAR)` returns **April and May 2023** — the same dates shifted back by one year. It preserves the exact date range structure, just offset in time. It does not return adjacent months (B), the full year (C), or a single month (D).

## Question 26

**Why must a date table be contiguous (no gaps) for time intelligence functions to work correctly in Power BI?**

A) Because DAX requires alphabetically sorted date columns  
B) Because time intelligence functions like TOTALYTD, SAMEPERIODLASTYEAR, and DATEADD generate date ranges based on sequential calendar logic — gaps would cause those ranges to miss values and return incorrect results  
C) Because the VertiPaq engine cannot compress non-contiguous date columns  
D) Because Power Query requires contiguous date ranges for proper type conversion  

**✅ Correct Answer: B**

**Explanation:** DAX time intelligence functions work by generating sets of dates (e.g., all dates from Jan 1 to the current date, or the same period last year). These functions assume that the date table contains **every calendar day** in the range. If there are gaps (e.g., weekends missing), the generated date sets may reference dates that don't exist in the table, causing measures to return incorrect or blank results. This is a fundamental requirement — not related to sorting (A), compression (C), or Power Query (D).

## Question 27

**A developer needs to create a category classification based on a product's list price: "Budget" (< $50), "Standard" ($50–$200), "Premium" (> $200). This classification needs to be usable as a slicer and in row-level security filters. Should they use a calculated column or a measure?**

A) Measure — because measures are evaluated at query time and can be used in slicers  
B) Calculated column — because it needs to exist as a physical column in the table for use in slicers and row filters  
C) Either — they are interchangeable for this scenario  
D) Neither — use a Power Query conditional column instead  

**✅ Correct Answer: B**

**Explanation:** **Calculated columns** are the correct choice when the value needs to act as a **table attribute** — usable in slicers, axes, legends, filters, and row-level security rules. A calculated column is computed once at refresh time and stored in the model for each row. **Measures** (A) are calculated dynamically per query context and cannot be used directly as slicer fields or in RLS expressions. They are not interchangeable (C). While a Power Query conditional column (D) is also valid and often preferred for better query folding, the question specifies a model-layer requirement — and between a calculated column and a measure, B is correct.

## Question 28

**When is a calculated table in DAX the preferred choice over a Power Query query?**

A) Always — calculated tables are faster than Power Query queries  
B) When the table logic depends on relationships in the data model (e.g., using RELATED(), FILTER(), or SUMMARIZE() across related tables), which cannot easily be reproduced in Power Query  
C) When the table needs to be refreshed independently  
D) When the table is used as an input to Power Query transformations  

**✅ Correct Answer: B**

**Explanation:** **Calculated tables** are defined using DAX expressions and are evaluated at model refresh time. They can reference other model tables, use relationship functions like `RELATED()`, and leverage the full DAX library (e.g., `SUMMARIZE`, `ADDCOLUMNS`, `CROSSJOIN`). This is their primary advantage — they can express logic that depends on the **data model's relationships**, which Power Query cannot access. For simple transformations and data shaping, Power Query (which supports query folding) is generally preferred. Calculated tables are not always faster (A), cannot be refreshed independently (C), and cannot feed back into Power Query (D).

## Question 29

**What is the key difference between a measure and a calculated column in terms of when they are evaluated?**

A) Both are evaluated at dataset refresh time  
B) Both are evaluated at report query time  
C) A calculated column is evaluated once at refresh/load time and stored per row; a measure is evaluated dynamically at query time based on the current filter context  
D) A calculated column is evaluated at report query time; a measure is evaluated at refresh time  

**✅ Correct Answer: C**

**Explanation:** **Calculated columns** are computed during the **data refresh** and their values are persisted in the VertiPaq engine for every row — they consume model memory. **Measures** are computed **on demand at query time**, evaluated fresh for every visual cell based on the current filter/row context. Measures do not consume model memory for stored values (only their expression). This is the fundamental distinction, affecting performance, memory, and usage patterns.

## Question 30

**A developer wants to calculate "Profit Margin %" as a value that changes based on the selected product, region, and time period in a report. Should they use a calculated column or a measure?**

A) Calculated column — because it involves division which is only supported in columns  
B) Measure — because the result needs to respond dynamically to filter context from slicers and visual filters  
C) Calculated table — because it involves multiple dimensions  
D) Quick measure — because profit margin is a predefined template  

**✅ Correct Answer: B**

**Explanation:** **Profit Margin %** is a ratio that should change as users interact with the report (selecting different products, regions, time periods). This is exactly what **measures** are designed for — they respond to the current filter context dynamically. A calculated column (A) would compute profit margin per row at a single granularity (e.g., per transaction) and cannot aggregate across different context combinations correctly. Division is supported in both columns and measures (A is doubly wrong). Calculated tables (C) are for generating tables, not scalar analytics. Quick measures (D) do have templates for ratios, but the core answer is that a measure is needed.

## Question 31

**A data model contains a "Sales" table and a "Budget" table that are not related to each other (both link to a shared Date dimension). A developer needs to create a table showing Sales vs Budget side by side for each month. What is the recommended approach?**

A) Create a direct many-to-many relationship between Sales and Budget  
B) Build a calculated table using SUMMARIZE or ADDCOLUMNS that joins both tables through the Date dimension  
C) Merge Sales and Budget in Power Query  
D) Use a UNION() calculated table  

**✅ Correct Answer: B**

**Explanation:** When Sales and Budget are both linked to a Date dimension but not to each other, a common approach is to use a **calculated table** built with `SUMMARIZE` or `ADDCOLUMNS` that brings together measures from both fact tables via the shared Date dimension. This creates a single tabular result for reporting. A direct M:M relationship (A) is not appropriate since these are separate fact tables, not a dimension relationship. Merging in Power Query (C) would require a join key and mixes transformation logic with model logic. `UNION()` (D) stacks rows, not merges measures from different tables.

## Question 32

**What is the difference between `COUNT` and `COUNTROWS` in DAX?**

A) COUNT counts the number of rows in a table; COUNTROWS counts non-blank values in a column  
B) COUNTROWS counts the number of rows in a table; COUNT counts non-blank values in a column  
C) They are identical and can be used interchangeably  
D) COUNT works only on text columns; COUNTROWS works only on number columns  

**✅ Correct Answer: B**

**Explanation:** `COUNTROWS(table)` counts the **number of rows in a table** (or table expression), including rows with blank values. `COUNT(column)` counts the number of **non-blank values in a column**. In practice, `COUNTROWS` is generally preferred for counting fact table rows because it's explicit and performs well. `COUNTA` counts non-blank values including text. Neither function is limited to specific data types (D is wrong). They are not interchangeable (C is wrong).

## Question 33

**A developer writes `[Distinct Customers] = DISTINCTCOUNT(Sales[CustomerID])`. What does this measure return?**

A) The total count of all CustomerID values, including duplicates  
B) The number of unique (distinct) CustomerID values in the Sales table within the current filter context  
C) The number of customers who made more than one purchase  
D) The number of CustomerID values that are not null  

**✅ Correct Answer: B**

**Explanation:** `DISTINCTCOUNT(column)` returns the number of **unique, distinct values** in the specified column within the current filter context. It ignores duplicates and counts each unique CustomerID only once. This is used for metrics like "number of unique customers who purchased." Total count including duplicates (A) would use `COUNT` or `COUNTROWS`. Customers with more than one purchase (C) requires additional logic. Non-null count (D) uses `COUNT` or `COUNTA`.

## Question 34

**Which DAX function returns the largest value in a column, ignoring blanks?**

A) `MAXA`  
B) `MAX`  
C) `LASTDATE`  
D) `TOPN`  

**✅ Correct Answer: B**

**Explanation:** `MAX(column)` returns the **largest value** in the specified column, ignoring blank values. It works with numeric, date, and text columns. `MAXA` (A) is similar to MAX but also evaluates logical values (TRUE/FALSE as 1/0). `LASTDATE` (C) returns the last date in the current filter context from a date column — it is a time intelligence function, not a general max function. `TOPN` (D) returns the top N rows of a table expression.

## Question 35

**A developer writes the measure `[Average Order Value] = AVERAGE(Sales[OrderAmount])`. A colleague suggests using `AVERAGEX(Sales, Sales[OrderAmount])` instead. In this specific case, what is the difference?**

A) AVERAGEX is faster because it uses iterators  
B) AVERAGE works on columns; AVERAGEX iterates over a table expression — for a simple column average they produce the same result, but AVERAGEX is more flexible for complex expressions  
C) AVERAGE ignores blank values; AVERAGEX includes them  
D) AVERAGEX can only be used with related tables  

**✅ Correct Answer: B**

**Explanation:** For a simple case of averaging a column, `AVERAGE(column)` and `AVERAGEX(table, expression)` **produce the same result**. The advantage of `AVERAGEX` is that the second argument is an **expression evaluated row by row**, allowing you to compute averages over complex calculations (e.g., `AVERAGEX(Sales, Sales[Quantity] * Sales[UnitPrice])`). `AVERAGE` can only reference a single column directly. Neither is universally faster (A — it depends on the scenario). Both ignore blanks by default (C is wrong). AVERAGEX can iterate any table (D is wrong).

## Question 36

**What does `SUMX(Sales, Sales[Quantity] * Sales[UnitPrice])` calculate?**

A) The sum of the Quantity column multiplied by the sum of the UnitPrice column  
B) The row-by-row product of Quantity and UnitPrice for each row in Sales, then the sum of all those products  
C) The average of Quantity times the average of UnitPrice  
D) The maximum product of Quantity and UnitPrice  

**✅ Correct Answer: B**

**Explanation:** `SUMX(table, expression)` is an **iterator** that evaluates the expression (`Sales[Quantity] * Sales[UnitPrice]`) for **each row** in the Sales table, then **sums** all those row-level results. This correctly calculates total revenue when price and quantity are in separate columns. Option A describes `SUM(Quantity) * SUM(UnitPrice)`, which would be mathematically incorrect for revenue. C and D describe average and max operations respectively.

## Question 37

**A developer needs a measure that counts the number of orders where the order amount exceeds $1,000. Which DAX expression is correct?**

A) `COUNTROWS(FILTER(Sales, Sales[OrderAmount] > 1000))`  
B) `COUNT(Sales[OrderAmount] > 1000)`  
C) `CALCULATE(COUNTROWS(Sales), Sales[OrderAmount] > 1000)`  
D) Both A and C produce the correct result  

**✅ Correct Answer: D**

**Explanation:** Both A and C are valid and equivalent approaches. **Option A** uses `FILTER` to reduce the Sales table to rows where OrderAmount > 1000 and then `COUNTROWS` to count them. **Option C** uses `CALCULATE` with a filter argument (`Sales[OrderAmount] > 1000`) to modify the filter context and then `COUNTROWS(Sales)` to count. Both produce the same result. `COUNT(Sales[OrderAmount] > 1000)` (B) is invalid syntax — COUNT takes a column reference, not a boolean expression.

## Question 38

**A developer writes `[Min Sale Date] = MIN(Sales[OrderDate])`. What does this return when no filters are applied?**

A) The first date in the Date dimension table  
B) The earliest OrderDate value across all rows in the Sales table  
C) Null, because MIN requires a filter context  
D) The minimum OrderDate for the current visual row's context  

**✅ Correct Answer: B**

**Explanation:** `MIN(column)` returns the **minimum value** in the specified column. Without any external filter context (e.g., no slicer or visual filter applied), it scans the entire Sales table and returns the **earliest (minimum) OrderDate** across all rows. It does not reference the Date dimension table (A). MIN works without a filter context (C is wrong). ## CALCULATE & Filter Context

## Question 39

**What is the primary purpose of the CALCULATE function in DAX?**

A) To perform arithmetic calculations like addition and multiplication  
B) To evaluate an expression in a modified filter context by adding, removing, or overriding existing filters  
C) To calculate running totals across rows  
D) To replace the need for relationships between tables  

**✅ Correct Answer: B**

**Explanation:** `CALCULATE(expression, filter1, filter2, ...)` is arguably the most important DAX function. Its primary purpose is to **evaluate an expression** (first argument) in a **modified filter context** established by the filter arguments. Filter arguments can add new filters, remove filters (using ALL, REMOVEFILTERS), or override existing ones. It is not a generic arithmetic function (A), does not compute running totals by default (C), and does not replace relationships (D).

## Question 40

**A developer writes:**
```dax
[All Products Sales] = CALCULATE([Total Sales], ALL(Products))
```
**What does this measure return?**

A) Total sales for all products, regardless of any product-related filter in the current context  
B) Total sales for the current product selection only  
C) Total sales divided by the number of products  
D) Total sales with all filters removed from every table  

**✅ Correct Answer: A**

**Explanation:** `ALL(Products)` removes all filters applied to the **Products** table. When used inside `CALCULATE`, it causes `[Total Sales]` to be evaluated as if no product filter is active — returning total sales across **all products**, even if a product slicer or visual row filter is applied. Filters from other dimensions (Date, Store, etc.) remain active. B describes normal [Total Sales] without ALL. C is a different calculation. D would use `ALL()` with no arguments or `REMOVEFILTERS()` on multiple tables.

## Question 41

**What is the difference between `ALL(Table)` and `ALLEXCEPT(Table, Column1)` in DAX?**

A) ALL removes all filters from the table; ALLEXCEPT removes all filters except the ones on the specified columns  
B) ALL removes filters from specified columns; ALLEXCEPT removes filters from the whole table  
C) They are identical but ALLEXCEPT is more efficient  
D) ALL works on tables; ALLEXCEPT works only on columns  

**✅ Correct Answer: A**

**Explanation:** `ALL(Table)` removes **all** filters from every column in the specified table. `ALLEXCEPT(Table, Col1, Col2, ...)` removes all filters from the table **except** the filters on the listed columns — those columns retain their filter context. This is useful for calculations like "% of parent category" where you want to remove the subcategory filter but keep the category filter. B has the definitions reversed. C is wrong — they have different behaviours. D is wrong — both can reference tables (the first argument is always a table).

## Question 42

**A developer creates the following measure:**
```dax
[Sales % of Total] = DIVIDE([Total Sales], CALCULATE([Total Sales], ALL(Sales)))
```
**When placed in a table visual with Product in rows, what does each row show?**

A) Total Sales for that product divided by Total Sales for the selected product  
B) Total Sales for that product divided by the grand total Sales across all products  
C) The percentage of sales above the average  
D) It always returns 1 (100%) because ALL cancels all filters  

**✅ Correct Answer: B**

**Explanation:** The denominator `CALCULATE([Total Sales], ALL(Sales))` evaluates `[Total Sales]` with all filters removed from the Sales table — this returns the **grand total** across all products. The numerator `[Total Sales]` is filtered by the current row's product. Dividing gives each product's share of the grand total. This is the standard "% of total" pattern. A is incorrect (denominator is grand total, not selected product total). C is a different calculation. D is wrong — `ALL(Sales)` removes filters from Sales in the denominator, but the numerator still has the product filter.

## Question 43

**What does `KEEPFILTERS` do when used as a filter argument in CALCULATE?**

A) It removes all existing filters from the filter context  
B) It ensures that the filter argument is applied as an additional intersection with existing filters, rather than overriding them  
C) It preserves filters from the row context only  
D) It is used to keep measures from being recalculated  

**✅ Correct Answer: B**

**Explanation:** By default, a filter argument in `CALCULATE` **overrides** any existing filter on the same column. `KEEPFILTERS(filter)` changes this behaviour so that the new filter is applied as an **intersection** with (not replacement of) the existing filter — meaning only rows satisfying both the existing filter AND the new filter are included. This is important when you want the filter argument to narrow down results rather than override them. A describes ALL/REMOVEFILTERS (opposite). C and D are incorrect descriptions.

## Question 44

**A developer writes the measure:**
```dax
[Prior Year Sales] = CALCULATE([Total Sales], SAMEPERIODLASTYEAR('Date'[Date]))
```
**What does SAMEPERIODLASTYEAR do to the filter context here?**

A) It adds the prior year's dates as an additional filter on top of the current date filter  
B) It replaces the current date filter context with the equivalent date range from one year earlier  
C) It removes all date filters and returns total historical sales  
D) It calculates the difference between current and prior year sales  

**✅ Correct Answer: B**

**Explanation:** `SAMEPERIODLASTYEAR('Date'[Date])` returns a table of dates shifted back one year. Inside `CALCULATE`, this table **replaces** the current date filter context on the Date table with the prior year's equivalent date range. For example, if the current context is March 2024, the prior year dates become March 2023. `[Total Sales]` is then evaluated within that modified date context. It does not add to existing filters (A). It does not remove all date filters (C). The measure returns the prior year value, not a difference (D).

## Question 45

**A sales table has columns "Quantity" and "UnitPrice". A developer needs to compute the total revenue. The table does NOT have a pre-calculated "Revenue" column. Which DAX measure is correct?**

A) `[Revenue] = SUM(Sales[Quantity]) * SUM(Sales[UnitPrice])`  
B) `[Revenue] = SUMX(Sales, Sales[Quantity] * Sales[UnitPrice])`  
C) `[Revenue] = CALCULATE(SUM(Sales[Quantity] * Sales[UnitPrice]))`  
D) `[Revenue] = SUM(Sales[Quantity] * Sales[UnitPrice])`  

**✅ Correct Answer: B**

**Explanation:** `SUMX(Sales, Sales[Quantity] * Sales[UnitPrice])` correctly iterates row by row, multiplying each row's Quantity by its UnitPrice, then summing all results. Option A multiplies the **total quantity** by the **total unit price** — mathematically wrong if different products have different prices. Option C and D use `SUM()` with an expression as an argument, which is invalid DAX syntax — `SUM` accepts only a column reference, not an expression.

## Question 46

**A developer needs to find the customer with the highest total purchases. Which DAX approach correctly identifies this in a measure context?**

A) `MAX(Sales[CustomerID])`  
B) `TOPN(1, VALUES(Sales[CustomerID]), [Total Sales], DESC)`  
C) `MAXX(Sales, [Total Sales])`  
D) `MAXX(SUMMARIZE(Sales, Sales[CustomerID], "CustomerSales", [Total Sales]), [CustomerSales])`  

**✅ Correct Answer: D**

**Explanation:** To find the maximum total sales **per customer**, you need to: (1) group by CustomerID and calculate each customer's total sales using `SUMMARIZE`, (2) apply `MAXX` to iterate over that summary table and find the maximum. Option D correctly does this. Option A returns the max CustomerID value (not a measure). Option B using TOPN returns a table of rows, not a scalar — not directly usable as a measure. Option C (`MAXX(Sales, [Total Sales])`) would evaluate `[Total Sales]` in the context of each Sales row, which returns the row-level amount, not the per-customer total.

## Question 47

**What is a "semi-additive measure" and which DAX pattern is typically used to handle it?**

A) A measure that can only be summed across time, not other dimensions — use TOTALYTD  
B) A measure that can be summed across some dimensions (e.g., products, regions) but not time — typically handled using LASTDATE or LASTNONBLANK for point-in-time snapshots  
C) A measure that requires a SUMX iterator to calculate correctly  
D) A measure that only works in DirectQuery mode  

**✅ Correct Answer: B**

**Explanation:** **Semi-additive measures** include metrics like inventory balance, account balance, or headcount — values that make sense to sum across products or stores but not across time (you don't add last month's headcount to this month's to get a year total; you want the latest value). The DAX pattern uses `CALCULATE([Measure], LASTDATE('Date'[Date]))` or `LASTNONBLANK` to return the **most recent value** in any time period. TOTALYTD (A) is for additive time measures. SUMX (C) is an iterator, not specific to semi-additive patterns. Semi-additive measures work in Import mode (D is wrong).

## Question 48

**A developer writes the measure:**
```dax
[End of Month Inventory] = CALCULATE([Inventory Balance], LASTDATE('Date'[Date]))
```
**When this measure is placed in a matrix with Year and Month in rows, what value appears for each month?**

A) The sum of all daily inventory values for that month  
B) The average daily inventory for that month  
C) The inventory balance as of the last date in that month  
D) The inventory balance as of the first date in that month  

**✅ Correct Answer: C**

**Explanation:** `LASTDATE('Date'[Date])` returns the **last date** in the current filter context. ## Question 53

**Which of the following practices best reduces the memory footprint of a Power BI dataset in VertiPaq (Import mode)?**

A) Using text data types for all columns to ensure flexibility  
B) Removing unused columns and tables, reducing cardinality of high-cardinality text columns, and using integer keys instead of long text strings in relationships  
C) Enabling bidirectional cross-filtering on all relationships  
D) Using measures instead of calculated columns for all computations  

**✅ Correct Answer: B**

**Explanation:** VertiPaq compresses data column by column using dictionary encoding and run-length encoding. The three most impactful optimisations are: (1) **Remove unused columns/tables** — every column occupies memory regardless of use. (2) **Reduce cardinality** — high-cardinality text columns (e.g., free-text descriptions, GUIDs) compress poorly; replacing them with integer lookups or categories dramatically reduces size. (3) **Use integer keys** for relationships instead of string keys. Using text types for everything (A) worsens compression. Bidirectional filtering (C) increases query complexity, not compression. Measures vs. calculated columns (D) affects runtime performance, not dataset size.

## Question 54

**What is the VertiPaq engine in Power BI, and what is its key characteristic?**

A) A query parsing engine that translates DAX into SQL  
B) An in-memory columnar storage engine that compresses imported data for extremely fast analytical query performance  
C) A caching engine that stores DirectQuery results temporarily  
D) The engine responsible for rendering visuals on the report canvas  

**✅ Correct Answer: B**

**Explanation:** **VertiPaq** (also known as the xVelocity engine) is the **in-memory columnar storage engine** that powers Power BI's Import mode. Data is compressed using dictionary encoding and run-length encoding, then stored column by column in RAM. This columnar structure enables extremely fast aggregation queries (SUM, COUNT, DISTINCTCOUNT) without scanning full rows. It is not a SQL translator (A — that is query folding in Power Query). It does not cache DirectQuery results for re-use across sessions (C). Visual rendering is handled by a separate frontend layer (D).
